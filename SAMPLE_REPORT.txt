# SAMPLE CODE4ARENA LAYERZERO AUDIT REPORT

## Title
[MEDIUM] ULN302 Receive Path: Nonce Tracking Race Condition Allows Message Replay and Double Execution

## Severity
**Medium**

---

## Summary

A race condition in the ULN302 receive message library's nonce validation allows an attacker to execute the same inbound message multiple times. The vulnerability exists in the gap between message verification (via DVN submission) and nonce increment, enabling double-processing of cross-chain transactions with potential for duplicate fund transfers, double state updates, and protocol invariant violations.

---

## Details

### Component Affected
- **Package**: `contracts/protocol/stellar/contracts/message-libs/uln-302`
- **File**: `src/receive_uln.rs`
- **Function**: `verify_and_commit_payload()` and `clear_nonce()` interactions
- **Lines**: Approximately lines 45-85 (core verification logic)

### Root Cause

The receive message verification process in ULN302 follows this sequence:

1. DVN submits verification (`submit_verification()`)
2. Message library checks if all required DVNs have verified
3. Nonce is marked as "committed" in storage
4. **Later**, OApp calls `Endpoint.clear()` which triggers message execution
5. Nonce is incremented to prevent replays

**The vulnerability**: Between steps 3 and 5, the message state is "verified but not yet incremented". If the OApp's `Endpoint.clear()` call can be triggered multiple times (or if another path exists), the message will be executed multiple times with the same nonce.

In pull-mode delivery on Soroban, the execute operation doesn't automatically increment nonce atomically with message execution because of reentrancy constraints. The endpoint calls the OApp's receiver, which could theoretically re-enter the endpoint with the same message before nonce is incremented.

### Code Evidence

**In `receive_uln.rs`**, the verify function (simplified):

```rust
pub fn verify_and_commit_payload(
    env: &Env,
    src_eid: u32,
    sender: &Bytes,
    payload_hash: &BytesN<32>,
) -> Result<(), Error> {
    // Check if all DVNs have verified
    let dvn_count = get_required_dvn_count(env, src_eid)?;
    let verified_count = get_verified_count(env, src_eid, sender, payload_hash)?;
    
    if verified_count < dvn_count {
        return Err(Error::InsufficientVerifications);
    }
    
    // Mark as committed
    set_committed_payload(env, src_eid, sender, payload_hash);
    
    // *** ISSUE: Nonce not incremented here ***
    // Nonce is only incremented in endpoint.clear() call later
    
    Ok(())
}
```

**In `endpoint_v2.rs`**, the clear function is called by OApp:

```rust
pub fn clear(
    env: &Env,
    src_eid: u32,
    sender: &Bytes,
    nonce: u64,
) -> Result<(), Error> {
    // Verify payload is committed (passed DVN verification)
    verify_payload_committed(env, src_eid, sender, nonce)?;
    
    // Execute the message (calls OApp receiver)
    let message = get_pending_message(env, src_eid, sender, nonce)?;
    execute_message(env, &message)?;
    
    // Only now is nonce incremented
    increment_nonce(env, src_eid, sender, nonce);
}
```

### Attack Scenario

**Preconditions**:
- All required DVNs have verified a message payload
- Message is committed in storage
- OApp's receiver contains code that makes external calls

**Attack execution**:

1. DVN1 submits verification for message M at nonce=100 to ULN302
2. DVN2 submits verification for message M at nonce=100 to ULN302
3. Both verifications complete; payload is marked as "committed" in ULN302
4. OApp calls `Endpoint.clear(src_eid, sender, nonce=100)`
5. Endpoint triggers OApp receiver contract via `execute_message()`
6. OApp receiver processes the message (transfers funds, updates state)
7. Before returning from step 5, the OApp receiver (or one of its callees) detects they can call `Endpoint.clear(src_eid, sender, nonce=100)` again
8. The second `clear()` call succeeds because:
   - Nonce hasn't been incremented yet (still at step 6)
   - Payload is still marked as committed
9. Message executes again, funds transferred twice, state corrupted

**Why this matters**: In Soroban's abstract account pattern with pull-mode delivery, the execution can be triggered from within the OApp receiver's context if there's any code path that calls back to the endpoint before the initial execution completes.

### Proof of Concept (Pseudocode)

```rust
// In test scenario
#[test]
fn test_message_double_execution_race_condition() {
    let env = Env::default();
    let endpoint = EndpointV2::new(&env);
    let uln = ULN302::new(&env);
    let oapp = TestOApp::new(&env);
    
    // Setup: Configure DVNs, OApp, etc.
    setup_protocol(&env, &endpoint, &uln, &oapp);
    
    // Step 1: Both DVNs verify the message
    uln.verify_payload(
        &env,
        src_eid,
        &sender_addr,
        &payload_hash,
        dvn1_sig,
    );
    uln.verify_payload(
        &env,
        src_eid,
        &sender_addr,
        &payload_hash,
        dvn2_sig,
    );
    
    // Step 2: First clear() call
    endpoint.clear(&env, src_eid, &sender_addr, nonce);
    assert_eq!(oapp.received_message_count(), 1); // Message received once
    
    // Step 3: Second clear() call - Should fail but doesn't due to race
    let result = endpoint.clear(&env, src_eid, &sender_addr, nonce);
    
    // VULNERABILITY: This succeeds instead of failing
    assert!(result.is_ok(), "Second clear() should have failed!");
    assert_eq!(oapp.received_message_count(), 2); // Message received TWICE!
}
```

## Recommended Fix

### Option 1: Atomic Nonce Increment (Recommended)

Increment nonce during commit phase, not during clear phase:

```rust
pub fn verify_and_commit_payload(
    env: &Env,
    src_eid: u32,
    sender: &Bytes,
    nonce: u64,
    payload_hash: &BytesN<32>,
) -> Result<(), Error> {
    // Check all DVNs verified
    let dvn_count = get_required_dvn_count(env, src_eid)?;
    let verified_count = get_verified_count(env, src_eid, sender, payload_hash)?;
    
    if verified_count < dvn_count {
        return Err(Error::InsufficientVerifications);
    }
    
    // Atomic: Mark committed AND increment nonce in same transaction
    set_committed_payload(env, src_eid, sender, payload_hash);
    increment_nonce(env, src_eid, sender, nonce); // Add this
    
    Ok(())
}
```

Then in `clear()`, verify nonce was already incremented:

```rust
pub fn clear(
    env: &Env,
    src_eid: u32,
    sender: &Bytes,
    nonce: u64,
) -> Result<(), Error> {
    // Check nonce already incremented (was verified and committed)
    verify_nonce_incremented(env, src_eid, sender, nonce)?;
    
    let message = get_pending_message(env, src_eid, sender, nonce)?;
    execute_message(env, &message)?;
    
    Ok(())
}
```

### Option 2: Use Storage-Based Lock

Implement a processing flag that prevents re-entry:

```rust
pub fn clear(
    env: &Env,
    src_eid: u32,
    sender: &Bytes,
    nonce: u64,
) -> Result<(), Error> {
    let processing_key = storage_key("processing", src_eid, sender, nonce);
    
    // Check if already processing
    if env.storage().get(processing_key).is_ok() {
        return Err(Error::AlreadyProcessing);
    }
    
    // Set processing flag
    env.storage().set(processing_key, &true);
    
    // Execute message
    let message = get_pending_message(env, src_eid, sender, nonce)?;
    execute_message(env, &message)?;
    
    // Increment nonce (now safe from re-entry)
    increment_nonce(env, src_eid, sender, nonce);
    
    // Clear flag
    env.storage().remove(processing_key);
    
    Ok(())
}
```
