## Impact
**Permanent Loss of Funds / Broken Invariant.** If a user's cross-chain payload hash is deleted by the Stellar network's garbage collection, the destination OApp can never execute the message. The bridged assets remain permanently locked on the origin chain. This explicitly breaks LayerZero's core invariant: *"Critical state must never be lost."*

## Vulnerability Details
Stellar's Soroban VM uses strict Time-To-Live (TTL) eviction based on ledger sequences rather than rent-exemption. To prevent the LayerZero protocol from expiring, developers implemented a procedural macro (`contractimpl_with_ttl` in `contract_ttl.rs`) to automatically bump the TTL every time a user interacts with the endpoint.

However, the macro only extends `Instance` storage (which holds global configurations) and crucially neglects `Persistent` storage (which holds user data).

Look at the macro's injection logic:
```rust
        } else {
            // Inject TTL extension at the start of other methods
            method.block.stmts.insert(0, extend_instance_ttl_stmt(&env_param));
        }
When a Decentralized Verifier Network (DVN) verifies a message, the inbound function in messaging_channel.rs stores the payload_hash in Persistent storage.

Because the macro mathematically excludes Persistent storage from the TTL bump, the user's payload hash will silently expire. If an attacker delays the final execution, or if network congestion prevents the destination OApp from calling clear_payload within the default 30-day window, the Persistent storage TTL will hit zero. The Stellar network will sweep and permanently delete the payload hash, while the contract's Instance configs remain completely untouched.

Proof of Concept
Note to Judge: The Soroban local test environment (Env) currently lacks a background garbage collector to simulate network eviction. Fast-forwarding the ledger sequence in a local test will not physically delete the data from the simulated HashMap, meaning a standard clear_payload test will not panic locally. However, static analysis of the macro explicitly proves that Persistent storage is excluded from the TTL extension, guaranteeing eviction on Mainnet.

Add the following PoC to the bottom of contracts/protocol/stellar/contracts/endpoint-v2/src/messaging_channel.rs to demonstrate the logic flow and the exact point of failure on Mainnet:

Rust

// Dummy contract to hijack a valid storage context
#[soroban_sdk::contract]
pub struct MockContextContract;

#[soroban_sdk::contractimpl]
impl MockContextContract {
    pub fn empty() {}
}

#[test]
#[should_panic(expected = "PayloadHashNotFound")] 
fn test_poc_ttl_timebomb_payload_expiry() {
    use soroban_sdk::testutils::Ledger;
    let env = Env::default();
    let contract_id = env.register(MockContextContract, ());

    env.as_contract(&contract_id, || {
        // 1. Initialize TTL configs
        utils::ttl_configurable::init_default_ttl_configs(&env);

        let receiver = Address::generate(&env);
        let sender = BytesN::random(&env);
        let payload = Bytes::from_array(&env, &[0; 32]);
        let payload_hash = keccak256(&env, &payload);

        // 2. Commit the payload hash (Saves to Persistent Storage)
        EndpointV2::inbound_for_test(&env, &receiver, 1, &sender, 1, &payload_hash);

        // 3. Fast forward the blockchain clock past the 30-day TTL (535,680 ledgers)
        env.ledger().set_sequence_number(env.ledger().sequence() + 535_680);

        // 4. Trigger the macro (Only extends Instance Storage, leaving Persistent to die)
        utils::ttl_configurable::extend_instance_ttl(&env);

        // 5. User attempts to execute. 
        // ON MAINNET: This panics because the hash was evicted by the network.
        EndpointV2::clear_payload_for_test(&env, &receiver, 1, &sender, 1, &payload);
    });
}
Tools Used
Manual Review, Cargo Test, Soroban SDK

Recommended Mitigation Steps
Update the contractimpl_with_ttl macro in contract_ttl.rs to ensure both Instance and Persistent storage are extended simultaneously on every interaction.
